# 10GB 以上大文件怎么传：断点下载、超时与网络质量检测

大文件传输失败，很少只是“文件太大”。更常见的原因是 Wi-Fi 抖动、浏览器休眠、磁盘空间不足、超时设置过短，或者下载完成时被系统安全策略拦截。

[狼享 LAN Share Lite](https://lan-share.whdatas.com/lite-version) 默认最大上传为 `10GB`，但可以调整。真正要做的是让应用参数、网络和存储同时匹配。

## 调整上传限制与超时

编辑 `app.yaml`：

```yaml
max_upload_size: "50GB"
write_timeout: 7200
read_timeout: 7200
```

- `max_upload_size` 是服务端接受的最大上传量；
- `write_timeout` 影响服务端向客户端下载时的写入时间；
- `read_timeout` 影响服务端读取上传请求的时间。

设置为更大并不等于一定更快，只是避免慢链路被过早终止。修改后需重启服务。

## 下载为什么更容易恢复

Lite 使用标准 HTTP 文件服务，支持 Range 请求。浏览器或下载工具可以按字节范围读取文件，这也是续传和拖动视频进度条的基础。

是否真正自动续传还取决于客户端：部分浏览器下载失败后不会自动恢复，专业下载工具通常更擅长。重要交付建议同时提供文件大小或校验值，方便确认完整性。

## 用连接质量检测做传输前判断

页面会通过小体积探测请求估算：

- 延迟；
- 抖动；
- 下载吞吐；
- 上传吞吐。

这不是标准实验室测速，结果主要用于回答：“当前浏览器到服务端的链路是否适合传大文件？”

如果延迟不高但抖动明显，页面浏览可能正常，大文件仍可能时好时坏。如果下载快、上传慢，通常是客户端 Wi-Fi 上行、路由器或终端性能限制。

## 网络选择优先级

对于几十 GB 文件，建议顺序是：

1. 千兆或更高速有线网络；
2. 同房间 5GHz/6GHz Wi-Fi；
3. 信号良好的普通 Wi-Fi；
4. 跨中继器、访客网络或拥挤无线网络。

服务端最好接有线，避免服务器和客户端同时争用无线信道。

## 磁盘与浏览器也会影响结果

传 50GB 文件前检查：

- 服务端目标磁盘剩余空间；
- 客户端下载目录剩余空间；
- 磁盘是否为慢速机械盘或 USB 2.0；
- 浏览器是否会因系统休眠暂停；
- 杀毒软件是否在完成时扫描大文件；
- Linux Snap 浏览器是否有下载目录权限。

遇到 Chrome 下载显示 `.crdownload` 或 `Unconfirmed`，不一定是服务端传输失败，也可能是浏览器完成重命名时被沙盒或安全软件拦截。

## 一个可复现的测试方法

不要直接用正式文件试错。先准备 1GB、5GB、10GB 三个测试文件，记录：

| 项目 | 内容 |
|---|---|
| 服务端 | CPU、内存、磁盘类型、连接方式 |
| 客户端 | 系统、浏览器、连接方式 |
| 网络 | 有线/Wi-Fi、链路速率 |
| 结果 | 耗时、平均速度、是否中断 |

只有写清测试环境，性能数字才有意义。

## 什么时候需要更专业的方案

Lite 适合可信网络中的大文件分发。如果需要多人并发配额、用户级限速、传输审计、告警和 SLA，Pro/Enterprise 更适合正式运营。
