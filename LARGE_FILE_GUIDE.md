# 📦 大文件传输指南

本文档说明如何使用 LAN Share 传输大文件（GB 级别），并解决传输过程中可能遇到的常见问题。

## ✅ 已实现的大文件支持

### 1. 上传优化

| 特性 | 说明 | 默认值 |
|------|------|--------|
| **最大上传大小** | 可配置的文件大小上限 | 10GB |
| **内存缓冲** | 使用 32MB 缓冲区，避免内存溢出 | 32MB |
| **临时文件** | 使用系统临时目录存储上传数据 | 自动 |
| **进度显示** | 实时显示上传进度和速度 | 自动 |

### 2. 下载优化

| 特性 | 说明 |
|------|------|
| **断点续传** | 支持 HTTP Range 请求，下载中断后可继续 |
| **流式传输** | 使用 32MB 缓冲区，边读边写 |
| **验证标识** | (开发中) 增加 ETag 与 Last-Modified 校验 |
| **Content-Length** | 正确设置文件大小，显示下载进度 |

### 3. 超时控制

| 参数 | 默认值 | 说明 |
|------|--------|------|
| **读取超时** | 3600 秒 (1 小时) | 客户端读取数据的超时时间 |
| **写入超时** | 3600 秒 (1 小时) | 客户端上传数据的超时时间 |

## 🚀 使用示例

### 传输大视频文件（5GB）

```bash
# 启动服务（默认支持 10GB）
./lan-share-lite -dir ~/Movies

# 在浏览器访问 http://<IP>:8080
# 拖拽 5GB 视频文件上传
# 实时显示进度：上传中：45.3% (2.3GB / 5.0GB)
```

### 传输超大文件（50GB）

```bash
# 增加上传限制并调整超时时间
./lan-share-lite -max-upload-size 50GB -write-timeout 7200
```

## 🔧 优化建议

### 1. 优先使用“文件名”下载
在大文件（如 ISO、4K 视频）下载时，建议 **直接点击文件名** 而非右侧的蓝色下载按钮。
*   **文件名下载**：触发 `http.ServeFile` 逻辑，协议支持更完备。

### 2. Ubuntu 用户的最佳选择
如果你在 Ubuntu 下遇到下载无法完成，请：
1.  改用 **Firefox** 或 **wget**。
2.  或者：安装 **.deb 原生版 Chrome**，避开 Snap 版的沙盒缺陷。

### 3. 网络稳定性
- **有线连接**：100-125MB/s（最推荐）。
- **5GHz WiFi**: 30-80MB/s（推荐）。
- **2.4GHz WiFi**: 5-15MB/s（不建议传输 >2GB 文件）。

### 3. 超时设置
如果网络较慢或文件超过 20GB，请务必增加超时时间：
```bash
# 设置为 4 小时超时
./lan-share-lite -write-timeout 14400 -read-timeout 14400
```

## 🛠️ 故障排查

### 问题 1: 下载显示 "Done" 但文件名是 `.crdownload` (Unconfirmed)

**可能原因**:
- **协议验证缺失**：服务端未提供 `ETag`，导致 Chrome 无法确认文件完整性。
- **Ubuntu 权限限制**：Snap 版 Chrome 运行在沙盒中，无权对下载目录的大文件进行重命名。
- **杀毒软件锁定**：Windows Defender 正在扫描大文件。

**解决方案**:
1.  **临时方案**：在 UI 界面直接点击文件名进行下载。
2.  **Ubuntu 用户**：经实测，**wget** 和 **Firefox** 在 Ubuntu 下均能正常完成 GB 级大文件的下载。如果 Chrome 出现 `Unconfirmed`，请改用这些工具：
    *   `wget http://<服务器IP>:<端口>/download/<文件名>`
3.  **Windows 用户**：稍等 1-2 分钟，待系统完成病毒扫描。

### 问题 2: 上传到一半失败

**可能原因**:
- 超过 `-max-upload-size` 限制。
- 磁盘空间不足（检查 `/tmp` 或目标挂载点）。
- 网络瞬间中断导致连接重置。
